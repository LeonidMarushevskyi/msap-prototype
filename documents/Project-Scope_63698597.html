Mississippi Agile Pool : Project Scope

Created by Leonid Marushevskiy, last modified on Oct 10, 2016

Deliverables
The deliverables for this effort are as follows:

  1. A working prototype hosted at a publicly-available URL
  2. A Git bundle of the entire repository, with all branches, including history
  3. A README.md file in the root directory of the repository that includes any instructions needed to get it up and running
  4. A brief description of the project, no greater than 150 words
  5. A detailed list of all artifacts used to create the prototype, including research findings, user stories, frameworks or libraries used and why, and functional descriptions for all components of the prototype. Place all documentation in the README.md file
  6. A detailed document of the team structure, including roles, responsibilities and corresponding Agile Labor Categories, assigning a team leader who will deliver the prototype
  7. Additional sufficient documentation that illustrates how to install and run the prototype on another target.

Prototype Requirements

Create a working prototype for the following user story:
“As a caseworker or parent, I would like to search for child care providers in my vicinity so that I can choose the best provider for a child.”

  1. Don't minify any of the JavaScript or CSS used in the application.
  2. The Vendor’s work should follow the principles and practices outlined in the U.S. Digital Services playbook*3. In addition to specific plays from the playbook outlined in this document, vendors are encouraged to incorporate as many plays as possible in their submission. When using a play from the handbook, please identify it in the artifacts and documentation that are submitted.
  3. Create a prototype that works on multiple devices (e.g. desktops, tablets, smart phones) using a responsive design
  4. Use modern and open-source technologies, regardless of architectural layer (frontend, backend, etc.). Prototype and underlying platforms used to create and run the prototype are openly licensed and free of charge

  5. Deploy the prototype on an Infrastructure as a Service (IaaS) or Platform as Service (PaaS) provider with
    5.1. the ability to easily manage the configuration.
    5.2. the ability to monitor the deployment and be alerted on issues.

  6. Develop automated unit tests for the code with adequate test coverage.
  7. Use a continuous integration system to automate the running of tests and to continuously deploy code to the IaaS or PaaS provider.

Documentation Requirements

  1. Write a brief description of the project, no greater than 150 words
  2. List out all artifacts used to create the prototype: including research findings, user stories, frameworks or libraries used and why, and functional descriptions for all components of the prototype.

Research and Discovery

Initiate a research phase to explore the existing tool and pinpoint the needs of the people who will use the service, and the ways the service will fit into their lives. Document the process and findings. (see: https://playbook.cio.gov/#play1

    1. Spend time with current and prospective users of the service.
    2. Use qualitative and quantitative research methods to determine people’s goals, needs, and behaviors; be thoughtful about the time spent.
    3. Test prototypes of solutions with prospective users you do not have a current direct relationship with (e.g. friends, coworkers, family, etc). Explain how you found them in the documentation.
    4. Document the findings about user goals, needs, behaviors, and preferences.
    5. Create a prioritized list of tasks the user is trying to accomplish.

Design

Initiate a research phase to explore and pinpoint the needs of the people who will use the service.

 

    1. Use a simple and flexible design style guide for the service (e.g. U.S. Web Design Standards*4).
    2. Give users clear information about where they are in each step of the process.
    3. Follow accessibility best practices to ensure all people can use the service.
    4. Use language that is familiar to the user and easy to understand.
    5. Use language and design consistently throughout the service, including online and offline touch points.
